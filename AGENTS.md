## Architecture Overview

This is a **multi-tenant SaaS** beauty salon management system (React + TypeScript + Vite) backed by **Supabase** (PostgreSQL, Auth, Realtime). It is deployed to Vercel as a PWA.

### Provider Stack (App.tsx)
The app wraps everything in a strict provider order:
```
QueryClientProvider → BrowserRouter → SupabaseAuthProvider
  → NotificationProviderAvancado → PWAProvider → Routes
```

### Authentication & Multi-Tenancy
- `src/contexts/AuthContext.tsx` is the real implementation; `SupabaseAuthContext.tsx` just re-exports it under `SupabaseAuthProvider` / `useSupabaseAuth`.
- On login, the user profile is fetched from the `usuarios` table and stored in context as a `Usuario` object (defined in `src/types/usuario.ts`).
- All Supabase tables have a `user_id` column scoped to the logged-in user. Row Level Security (RLS) policies (in `supabase_rls_policies.sql` / `supabase_multi_tenant_rls.sql`) enforce isolation automatically.
- The theme (`feminino` | `masculino`) is applied as a `data-theme` attribute on `<html>` and stored in `localStorage`.

### Two Supabase Clients
- `src/integrations/supabase/client.ts` – authenticated client with session persistence; used everywhere in the admin app.
- `src/integrations/supabase/publicClient.ts` – anonymous client with no session; used only by the public booking page (`/agendamento-online`) so that unauthenticated visitors can read services/schedules.

### Data Layer Pattern (Hooks)
`src/hooks/useSupabaseBase.ts` provides a generic `useSupabaseBase<T>(tableName)` hook with:
- `loadData()` – SELECT filtered by `user_id`
- `createRecord()` / `updateRecord()` / `deleteRecord()` – mutations with toasts
- `setupRealtimeSubscription()` – Supabase Postgres CDC channel

Every domain hook (`useSupabaseAgendamentos`, `useSupabaseClientes`, `useSupabaseServicos`, etc.) builds on top of this base. When adding a new data entity, create a `useSupabase{Entity}.ts` in `src/hooks/` following the same pattern.

### Subscription & Access Control
Users have `subscription_status` (`trial | active | expired | inactive`) and `plan_type` (`mensal | vitalicio`). The `TrialBanner` component is shown inside the `Layout` when status is `trial`. Trial blocking is enforced at the database level via `enforce_trial_block.sql`. Payment goes through Cakto (see `/checkout`, `/pagamento/retorno`, `/integracao-cakto`).

### Routing
- **Public** (no auth required): `/login`, `/cadastro`, `/agendamento-online` (also `/agendar`, `/agendamento-publico`), `/planos`, `/checkout`, `/pagamento/retorno`, `/privacidade`, `/termos`, `/sobre`.
- **Protected** (wrapped in `<ProtectedRoute>` + `<Layout>`): `/` (Dashboard), `/minha-agenda`, `/clientes`, `/servicos`, `/cronogramas`, `/financeiro`, `/configuracoes`, `/auditoria`, `/marketing`, `/produtos`, `/assinatura`, `/integracao-cakto`.
- `/agendamentos` → redirects to `/minha-agenda?tab=lista`; `/agenda` → `/minha-agenda?tab=semana`.
- **Add new routes above the `path="*"` catch-all** in `src/App.tsx`.

### Path Alias
`@/` resolves to `src/`. Use this for all internal imports.

### UI & Styling
- Components are from **shadcn/ui** (`src/components/ui/`), configured in `components.json`.
- Custom salon colors (`lilac`, `lavender`, `pink-accent`) are defined as CSS variables in `src/index.css` and extended in `tailwind.config.ts`. Use these tokens rather than hardcoded hex values.
- Responsive utilities (`container-responsive`, `p-responsive-sm`, `text-responsive-*`, `btn-touch`) are defined as Tailwind utilities in `src/index.css`.

### PWA
The service worker is a custom file at `src/service-worker.js`, managed by `vite-plugin-pwa` using the `injectManifest` strategy. Do not manually register it — the plugin handles registration.

### Database Schema
SQL schema and migration files are in the project root (e.g., `supabase_complete_schema.sql`, `supabase_rls_policies_complete.sql`). The auto-generated TypeScript types for the full database schema are at `src/integrations/supabase/types.ts` — do not edit this file manually.
